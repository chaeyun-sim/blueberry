# SPEC-TIMER-001 인수 기준 (Acceptance Criteria)

---

## REQ-1: 스톱워치 기본 동작

### 시나리오 1-1: 타이머 정상 시작

```
Given  의뢰 상태가 "진행 중(in_progress)"인 CommissionDetail 페이지가 열려 있고
       스톱워치 위젯의 시작 버튼이 활성화 상태이다
When   사용자가 시작 버튼을 클릭한다
Then   타이머 디스플레이가 "00:00:00"에서 카운트업을 시작하고
       1초 이내에 "00:00:01"로 갱신되며
       시작 버튼이 정지 버튼으로 전환된다
```

### 시나리오 1-2: 타이머 정상 정지

```
Given  스톱워치가 실행 중이고 디스플레이에 "00:02:34"가 표시되어 있다
When   사용자가 정지 버튼을 클릭한다
Then   타이머 디스플레이가 "00:02:34"에서 멈추고
       정지 버튼이 시작 버튼으로 전환되며
       저장 중 로딩 인디케이터가 표시된다
```

### 시나리오 1-3: 타이머 갱신 지연 성능

```
Given  스톱워치가 실행 중이다
When   타이머가 동작하는 동안
Then   UI 갱신은 매 1초 이내에 이루어지며
       경과 시간 표시 오차는 실제 경과 시간 대비 100ms 이내이다
```

### 시나리오 1-4: 다른 의뢰에서 타이머 실행 중 충돌 방지

```
Given  의뢰 A의 타이머가 실행 중이다
When   사용자가 의뢰 B의 상세 페이지로 이동하여 시작 버튼을 클릭한다
Then   의뢰 A의 타이머가 자동으로 정지되고 세션이 저장되며
       의뢰 B의 타이머가 새로 시작된다
       사용자에게 "이전 의뢰의 타이머가 자동 저장되었습니다" 토스트가 표시된다
```

## REQ-2: 시간 기록 저장

### 시나리오 2-1: 세션 정상 저장

```
Given  스톱워치가 실행 중이고 "00:05:00"이 경과되었다
When   사용자가 정지 버튼을 클릭한다
Then   Supabase `commission_time_logs` 테이블에
       - commission_id: 현재 의뢰 ID
       - user_id: 현재 사용자 ID
       - started_at: 타이머 시작 시각 (ISO 8601)
       - ended_at: 타이머 정지 시각 (ISO 8601)
       - duration_ms: 300000 (5분 = 300,000ms)
       레코드가 삽입된다
```

### 시나리오 2-2: 누적 시간 갱신

```
Given  의뢰의 기존 누적 시간이 "1시간 30분"이고
       새로운 세션 5분이 저장된다
When   세션 저장이 완료된다
Then   `commissions.total_time_spent_ms`가 5,700,000ms(1시간 35분)로 갱신되고
       스톱워치 위젯의 "누적: 1시간 35분" 표시가 업데이트된다
```

### 시나리오 2-3: 저장 실패 시 재시도

```
Given  사용자가 정지 버튼을 클릭하고 네트워크 오류가 발생한다
When   Supabase 저장 요청이 실패한다
Then   "작업 시간 저장에 실패했습니다. 다시 시도해주세요" 토스트가 표시되고
       세션 데이터가 localStorage에 보관되며
       타이머 위젯에 재시도 버튼이 나타난다
```

### 시나리오 2-4: 완료 상태 전환 시 자동 정지

```
Given  스톱워치가 "00:10:00" 실행 중이고
       사용자가 의뢰 상태를 "완료(done)"로 변경한다
When   상태 변경 요청이 Supabase에 성공한다
Then   타이머가 자동으로 정지되고
       현재 세션(10분)이 자동으로 저장되며
       "타이머가 자동 저장되었습니다" 토스트가 표시된다
```

## REQ-3: 작업 이력 조회

### 시나리오 3-1: 누적 작업 시간 표시

```
Given  의뢰의 `total_time_spent_ms`가 5,400,000ms(1시간 30분)이다
When   사용자가 CommissionDetail 페이지를 연다
Then   스톱워치 위젯에 "누적: 1시간 30분"이 표시된다
```

### 시나리오 3-2: 세션 이력 펼침 조회

```
Given  해당 의뢰에 3개의 시간 로그가 존재한다
When   사용자가 "이력 보기" 버튼을 클릭한다
Then   3개의 세션이 날짜 내림차순으로 표시되고
       각 항목은 시작 시각, 종료 시각, 소요 시간을 포함한다
       예: "2026-03-04  14:00 ~ 15:30  (1시간 30분)"
```

### 시나리오 3-3: 이력 없음 안내

```
Given  해당 의뢰에 시간 로그가 한 건도 없다
When   사용자가 "이력 보기" 버튼을 클릭한다
Then   "아직 기록된 작업 시간이 없습니다" 안내 메시지가 표시된다
```

## REQ-4: 상태별 타이머 연동

### 시나리오 4-1: 활성 상태에서 시작 버튼 활성화

```
Given  의뢰 상태가 "대기(pending)" 또는 "진행 중(in_progress)"이다
When   사용자가 CommissionDetail 페이지를 연다
Then   스톱워치 시작 버튼이 활성화(enabled)되어 있다
```

### 시나리오 4-2: 완료 상태에서 시작 버튼 비활성화

```
Given  의뢰 상태가 "완료(done)"이다
When   사용자가 CommissionDetail 페이지를 연다
Then   스톱워치 시작 버튼이 비활성화(disabled)되어 있고
       버튼에 마우스를 올리면 "완료된 의뢰는 타이머를 사용할 수 없습니다" 툴팁이 표시된다
```

### 시나리오 4-3: 취소 상태에서 시작 버튼 비활성화

```
Given  의뢰 상태가 "취소(cancelled)"이다
When   사용자가 CommissionDetail 페이지를 연다
Then   스톱워치 시작 버튼이 비활성화(disabled)되어 있고
       툴팁이 표시된다
```

## REQ-5: 로컬 상태 지속성

### 시나리오 5-1: 새로고침 후 타이머 복원

```
Given  의뢰 A의 타이머가 5분간 실행 중이고
       사용자가 브라우저를 새로고침한다
When   의뢰 A의 CommissionDetail 페이지가 다시 로드된다
Then   타이머가 5분 + 새로고침 소요 시간에서 이어서 실행되고
       디스플레이에 실제 경과 시간이 표시된다
       (오차 허용 범위: 2초 이내)
```

### 시나리오 5-2: 다른 페이지 이동 후 복원

```
Given  의뢰 A의 타이머가 실행 중이고
       사용자가 다른 페이지(예: Files)로 이동했다가 돌아온다
When   의뢰 A의 CommissionDetail 페이지로 재진입한다
Then   타이머가 이탈 전 경과 시간 + 이탈 중 경과 시간을 합산하여 계속 실행된다
```

### 시나리오 5-3: 유효하지 않은 localStorage 상태 처리

```
Given  localStorage에 삭제된 의뢰 ID를 가리키는 타이머 상태가 저장되어 있다
When   앱이 부팅된다
Then   해당 localStorage 항목이 자동으로 삭제되고
       타이머 복원 시도 없이 정상 상태로 렌더링된다
```

## 성능 기준

| 항목 | 기준값 | 측정 방법 |
|------|--------|----------|
| 타이머 UI 갱신 지연 | < 100ms | `performance.now()` 기반 tick 간격 측정 |
| setInterval drift 누적 오차 | < 2초 / 10분 | 10분 실행 후 실제 경과 시간과 비교 |
| 세션 저장 응답 시간 | < 2,000ms (P95) | Supabase 응답 시간 기준 |
| 페이지 로드 시 타이머 복원 | < 500ms | localStorage 읽기 + 스톱워치 재시작 |

## 품질 기준

| 항목 | 기준 |
|------|------|
| TypeScript 에러 | 신규 파일 포함 0개 (`tsc --noEmit` 통과) |
| 메모리 누수 | `useEffect` cleanup에서 `clearInterval` 호출 필수. React DevTools Profiler로 검증 |
| 컴포넌트 언마운트 | 위젯 언마운트 시 setInterval 정리, localStorage 상태 유지 (타이머 실행 중이면 유지) |
| RLS 정책 | `commission_time_logs` 테이블은 `auth.uid() = user_id` 정책으로 본인 데이터만 접근 |
| 접근성 | 시작/정지 버튼에 `aria-label` 명시 ("타이머 시작", "타이머 정지") |
